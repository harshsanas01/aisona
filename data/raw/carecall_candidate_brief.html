CareCall Insight

Full-stack coding exercise — build a question-answering tool over care-call transcripts

Time limit: 90 minutes · Any language, framework, or stack · AI coding assistants allowed · 30-minute debrief follows · Kit v1.2

Read this first. This exercise is deliberately bigger than 90 minutes. You are not expected to finish everything — what you choose to build first, what you consciously cut, and how you explain those choices are evaluated as much as the code itself. Get the core loop working end-to-end, then improve whatever you think matters most.

0. Before the session — environment checklist

The 90 minutes start with the kit in your hands; environment setup is not part of the exercise. Verify the day before:

  • Your editor/IDE and the AI coding tools you plan to use are installed and logged in.
  • A language runtime you can start a web server with — confirm a hello-world page serves locally.
  • git is configured (git init && git commit works with your identity).
  • A handover path: a repo host you can push to, or the ability to zip the project folder.
  • If you want to use an LLM API: have your own key ready, or request the temporary key before the session starts.

If your environment breaks mid-session, say so immediately instead of silently debugging it — the interviewer decides how to handle lost time.

1. Scenario

Harborview Senior Day Center (fictional) runs an AI phone assistant named Amy that makes short daily check-in calls to older adults in its day program. The care staff can't listen to every call, but the calls are full of things they need to know — new medications, dizzy spells, missed meals, ride complaints.

Your job: build a small internal web tool that lets a care coordinator ask questions across the transcripts in natural language — "Who mentioned feeling dizzy this month?" — and get trustworthy, source-linked answers.

2. What you're given

  • carecall_transcripts.json — 21 check-in calls (8–13 turns each) with patient name, date, and per-turn speaker labels.
  • carecall_questions.json — 8 sample questions with the calls a good answer should be grounded in. Use them to test yourself. Some are intentionally unanswerable from the data.
  • LLM access — three equally acceptable options:
    1. your own API key (any provider),
    2. a temporary rate-limited key from us — just ask,
    3. mock mode: no LLM at all; return the retrieved passages as an extractive answer.
    Mock mode is not penalized — we grade the pipeline and the grounding, not the prose. Whichever you choose, make the LLM component swappable.
Scale note (this matters): the 20 calls are a small sample. The production corpus is ~50,000 calls, growing by ~2,000 per week. Your design should not assume the whole corpus fits into a single prompt. Shortcuts are fine — as long as you document them and know what you'd change.

Transcript schema

{
  "calls": [
    {
      "call_id": "call_001",
      "date": "2026-05-26",
      "patient": { "id": "P-1003", "name": "Rosa Kim", "age": 75 },
      "duration_seconds": 412,
      "turns": [
        { "speaker": "assistant",   "text": "Good morning, Rosa! ..." },
        { "speaker": "participant", "text": "Oh, good morning, Amy. ..." }
      ]
    }
  ]
}

3. Core requirements — build these first

  1. C1 Backend service that loads and indexes the transcripts and exposes a question-answering endpoint. Every answer must cite its sources — the call ID(s) and the specific turns the answer is based on. If the data doesn't contain an answer, the system should say so — not guess.
  2. C2 Web UI: type a question → see the answer with its cited sources; click a source → view that transcript.
  3. C3 Retrieval that scales in principle — see the scale note above. Any retrieval method is fine (keyword, TF-IDF/BM25, embeddings, hybrid) if you can explain why you chose it.

4. Optional extensions — pick at most 1–2, only after the core works

  • Filter questions by patient and/or date range
  • Automated eval script against carecall_questions.json (e.g., report retrieval hit rate)
  • Highlight safety-relevant utterances (dizziness, near-misses, missed medications…) in the transcript view
  • Streaming answers
  • Hybrid search (keyword + semantic)
  • One-command run (e.g., docker-compose)
  • Endpoint to ingest a newly completed call transcript

5. Explicit non-goals

Do not spend time on: authentication, deployment, mobile layouts, or pixel-perfect visual design. Time spent here is wasted.

6. Rules

  • Any language, framework, database, or library. Vector DBs allowed if you can say why.
  • AI coding assistants are allowed and encouraged. But you must understand every line you submit — the debrief includes a short hands-on check of your own code.
  • No secrets committed to the repo (use environment variables).
  • Commit as you go. We read the commit history to see how you worked.

7. Deliverables

  • A git repository (link or zip) with your real commit history.
  • A README containing:
    • how to run it (≤ 3 commands),
    • a short architecture overview (a small diagram or ~5 lines),
    • key decisions and trade-offs (what you cut and why),
    • what you would do next with one more day,
    • one sentence on where AI tools helped you most.

8. How we evaluate

Six dimensions — none of them requires completeness:

DimensionWhat we look at
Scoping & judgmentDid the core loop come first? Were cuts deliberate and explained?
ArchitectureSeparation of concerns; is the LLM/retrieval swappable; where would this design bend at scale?
Backend pipelineIngestion → indexing → query path; error handling; citations traced from actual retrieval.
Retrieval & LLM literacyChunking choices for dialogue, retrieval method rationale, grounding, behavior on unanswerable questions.
FrontendA usable ask→answer→source loop; loading/error/empty states; sources you can actually open.
CommunicationREADME honesty and clarity; commit history; how you explain decisions in the debrief.

A smaller thing that works, cleanly structured and honestly documented, beats a bigger thing that's broken.

9. Debrief (30 min, right after the build)

We'll spend the session on four things: a demo where you show us how a care coordinator would actually use your tool; your architecture choices and what you'd change at scale; your retrieval & LLM decisions (chunking, ranking, grounding, evaluation); and the design and thoroughness of your pipeline (failure paths, citations, edge cases). We'll finish with a brief hands-on check of your own code — not a line-by-line recital, just enough to see that it's yours.

10. FAQ

  • Can I use embeddings / a vector DB? Yes — if you can say why. TF-IDF with a good justification scores as well as embeddings with none.
  • Can the answer be extractive (no LLM)? Yes. Mock mode is a first-class option.
  • What if I run out of time? Write what's missing and how you'd build it in the README. Unfinished-but-explained beats finished-but-unexplainable.
  • Can I look things up online? Yes — docs, Stack Overflow, AI chat, anything. Same rule applies: you own every line.
